# AGENTS

Этот документ помогает автономным агентам программирования работать в данном репозитории.


## О проекте

Это рабочее пространство используется сотрудником ИТ-компании. Оно содержит инструменты, документацию, заметки и репозитории.
Поддержка ИИ интегрирована через конфигурации Opencode, OpenClaw, bai и MCP для запуска локальных инструментов (Jira, GitLab, Pandoc, Bitrix24 и т. д.).
Этот проект работает в DevContainer, регулярно пересобирается. Все постоянные файлы подключаются к контейнеру с хоста, править их можно на хосте, можно в devcontainer. 

## Обзор репозитория

1. Корень репозитория смонтирован по пути `/workspace` в DevContainer.
   - `/workspace/.devcontainer/compose.yaml` — описывает архитектуру проекта, запущенные сервисы, в том числе DevContainer.
   - `/workspace/.devcontainer/Dockerfile` — описывает сборку DevContainer.
   - `/workspace/.opencode/opencode.json` — конфигурация OpenCode.
   - `/workspace/openclaw` — конфигурация OpenClaw.
   - `/workspace/.devcontainer/.env` — переменные окружения, секреты, параметры настройки.
   - `/workspace/.devcontainer/.env-dist` — шаблон переменных окружения.
   - `/workspace/.config/*.template` — шаблоны конфигурации, которые при сборке превращаются в конфигурационные файлы внутри контейнера подстановкой переменных окружения.
2. Загруженные и выгруженные файлы хранятся в `~/downloads`, bind mount к текущему devcontainer.
3. Рабочие заметки в формате Org-mode находятся в `~/org/work`. Новые задачи следует помещать в `~/org/inbox.org`, ставить TODO и сегодняшнюю дату.
   В задачу прикладывай ссылку на источник: обсуждение в mattermost, telegram, задачу в Jira или страницу Confluence. Далее оттуда пользователь обработает их вручную.
4. Другие репозитории расположены в `~/work`.
5. В корне рабочего пространства отсутствуют специфичные для языков манифесты сборки/линтинга/тестирования (например, `package.json`, `pyproject.toml`).

## Команды сборки

Этот проект работает в постоянном DevContainer. Все изменения окружения должны быть «запечены» в процесс сборки, чтобы сохраняться после пересборки.

**Процесс интеграции зависимостей:**
1. При изменении окружения, установке библиотек и т. д. сначала установите их внутри контейнера, проверьте функциональность, а затем добавьте в общую сборку.
2. Просите пользователя добавить новые ключи/секреты в `.devcontainer/.env`. Самостоятельно смотреть в файл `.devcontainer/.env`` нельзя.
3. Для проверки наличия переменных в `.env` вызывай `make show-env` и проверяй названия в `.devcontainer/show-env`.
4. Пользователь запускает `.devcontainer/build`, `.devcontainer/up` чтобы внедрить изменения в Docker-образ.
5. Пользователь запускает `.devcontainer/activate`, чтобы пересобрать контейнер и подключиться к обновленной сессии.

### 2) Команды MCP (Разработка)

Локальные/удаленные подключения MCP определены в `.opencode/opencode.json`.

Ключевые команды из `opencode.json`:

- Jira и Confluence (локально):
  - `uvx mcp-atlassian`
- Pandoc MCP (локально):
  - `uvx mcp-pandoc`
- Bitrix24 MCP (локально):
  - `uvx bitrix24-mcp`
- Mattermost MCP (локально):
  - `mattermost-mcp`
- GitLab (только чтение):
  - `npx -y @zereight/mcp-gitlab`
- Docling (удаленный MCP-сервер):
  - Контейнер запущен по адресу `http://docling-serve:5002/mcp`
  - Предоставляет инструменты конвертации и манипуляции документами (конвертация, экспорт в markdown и т. д.)
  - См. конфигурацию контейнера в `.devcontainer/compose.yaml`

## Рекомендации по стилю кода

### Общие (применимо ко всем языкам)
1. Делайте изменения минимальными и локализованными.
2. Предпочитайте читаемость излишней изобретательности.
3. Избегайте дублирования логики; выносите её только тогда, когда это значимо снижает сложность.
4. Делайте обработку ошибок явной на границах интеграции (запуск процессов, сетевые вызовы, парсинг внешних данных).
5. Никогда не фиксируйте (commit) секреты. Используйте переменные окружения и плейсхолдеры `{env:VAR}`, где это поддерживается.
6. Используйте символы ASCII в коде и документации, если в репозитории уже не используется другой стандарт.

### Импорты / Зависимости
1. Импортируйте только то, что используете.
2. По возможности предпочитайте стабильные, зафиксированные (pinned) версии зависимостей.
3. Для инструментов времени выполнения, указанных в `opencode.json`, сохраняйте массивы команд (`command: [...]`) детерминированными.
4. Не добавляйте неиспользуемые зависимости; обновляйте манифесты стилей/блокировок (lock-файлы), если они существуют в целевом подпроекте.

### Форматирование
1. Соблюдайте существующее форматирование в изменяемых файлах.
2. Сохраняйте разумную длину строки (обычно <= 100 символов, если в репозитории не установлено иное правило).
3. Используйте последовательные отступы (2 пробела в JSON/YAML; 4 пробела для блоков Python, где это применимо).
4. JSON: по возможности сохраняйте стабильный порядок ключей.

### Соглашения об именовании
1. Предпочитайте описательные имена аббревиатурам (за исключением общепринятых, таких как `id`, `url`, `env`).
2. Функции: `camelCase` (JS/TS) или `snake_case` (Python) в зависимости от экосистемы.
3. Константы: `UPPER_SNAKE_CASE` (Python) или `SCREAMING_SNAKE_CASE` (где это принято).
4. Избегайте двусмысленных имен, таких как `data`, `info`, `temp`.

### Обработка ошибок
1. Границы взаимодействия с сетью или процессами должны обрабатывать:
   - таймауты
   - коды выхода, отличные от 2xx или нуля
   - некорректно сформированные ответы
2. Включайте в ошибки достаточно контекста для отладки без необходимости повторного запуска с дополнительным логированием.
3. Для повторных попыток (retries) ограничивайте число попыток и выводите финальную ошибку.
4. Убедитесь, что логи не содержат секретов (токены, пароли, персональные ключи доступа).

## Операционные заметки для агентов
1. Если вам нужно запустить код репозитория, сначала найдите поддиректории с их собственными конфигурациями сборки/тестирования.
2. При редактировании `opencode.json` сохраняйте общую структуру схемы и оставляйте все плейсхолдеры окружения в формате `{env:...}`.
3. При добавлении новых команд в `AGENTS.md` включайте точные фрагменты кода оболочки (shell snippets), которые агенты могут скопировать и вставить.
